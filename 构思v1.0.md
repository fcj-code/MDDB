# Markdown 数据库架构设计 — 构思 v1.0

> 基于与 AI 助手的深入对话整理，探讨以 Markdown 文件为底层存储的数据库系统架构设计。

---

## 一、核心架构设计（四层架构）

### 第一层：存储引擎层

- **块（Block）存储引擎**：将每个 `.md` 文件解析为可寻址的"块"（段落、列表项、代码块等），每个块分配唯一 ID（文件路径:行号:哈希值），支持块级 CRUD
- **WAL 预写日志**：所有修改先写日志再同步到 `.md` 文件，保证事务回滚和崩溃恢复
- **版本控制集成**：自动 Git 提交，支持时间旅行查询
- **冲突解决**：基于 OT（操作转换）或 CRDT 的多人并发编辑处理

### 第二层：索引与查询层

- **多级索引系统**：文件级索引 → 块级 B+树索引 → 属性倒排索引 → 全文索引（Elasticsearch/Lucene）→ 向量索引 → 图谱索引
- **查询处理器**：扩展 SQL/DQL 方言，支持自然语言查询和图查询语言
- **查询优化器**：基于成本选择索引、查询重写优化、并行执行计划

### 第三层：事务与并发层

- **MVCC 事务管理器**：读不阻塞写，支持读已提交、可重复读隔离级别
- **锁管理**：文件级锁 + 块级锁 + 基于内容哈希的乐观锁
- **两阶段提交**：跨多个 `.md` 文件的事务原子性保证

### 第四层：计算与接口层

- **流处理引擎**：实时监听文件变化，触发工作流
- **批处理引擎**：夜间批量统计、生成报告
- **图计算引擎**：社区发现、影响力分析
- **多接口**：Obsidian 插件 API、RESTful API、GraphQL API、WebSocket 实时同步

## 二、三种数据记录方案

| 方案 | 记录粒度 | 语法格式 | 适用场景 |
|------|----------|----------|----------|
| **方案一** | 列表项/段落 | `姓名:: 张三 \| 公司:: 腾讯` | 最贴近 Dataview 风格，数据与备注共存 |
| **方案二（推荐）** | 每行 | `张三 \| 腾讯 \| 产品总监 \| 2023-08-15` | 分隔符结构化行，严格字段，适合批量操作 |
| **方案三** | 每个文件 | YAML Frontmatter | Obsidian Bases 原生支持，适合主数据 |

**方案二是最终共识**，其格式为：

- HTML 注释块定义表元数据（字段名、类型、约束）
- 每行一条记录，用 `|` 分隔字段
- 支持 `relation(table)` 类型实现外键关联
- 支持 JOIN、GROUP BY、聚合等类 SQL 查询

**示例文件格式：**

```markdown
<!--
schema_version: 1.0
table_name: contacts
fields: 姓名 | 公司 | 职位 | 相识日期 | 重要度 | 最后联系 | 标签
types: string | string | string | date | enum(高,中,低) | date | array
required: true | true | false | true | false | false | false
-->
张三 | 腾讯 | 产品总监 | 2023-08-15 | 高 | 2024-05-10 | #技术 #重要
李四 | 字节跳动 | 后端架构师 | 2024-01-20 | 中 | 2024-06-01 | #技术
王五 | 创投基金 | 投资合伙人 | 2022-11-03 | 高 | 2024-04-15 | #融资 #重要
```

## 三、字段数据类型规范（四阶段演进）

| 阶段 | 内容 | 具体类型 |
|------|------|----------|
| **阶段1：基础类型** | 标量 + 时间 + 枚举 + 数组 | `string`, `integer`, `float`, `boolean`, `date`, `datetime`, `enum`, `array` |
| **阶段2：高级类型** | Markdown 特有类型 | `reference`（跨表引用）, `markdown`, `code`, `link`, `tags` |
| **阶段3：增强类型** | 语义更精确 | 联合类型 `string\|integer`，可选类型 `string?`，字面量类型 `"high"\|"medium"\|"low"` |
| **阶段4：验证引擎** | 完整验证链 | 类型验证 → 约束验证 → 自定义验证 → 跨字段验证 |

### 类型系统设计原则

- **文本友好**：所有类型最终可序列化为纯文本，保证双向无损转换
- **渐进增强**：允许弱类型（自动推断）和强类型（显式声明）共存
- **声明式约束**：可组合的内置约束 + 自定义验证器

## 四、视图层设计（八大基础视图）

| 视图 | 用途 | 核心特性 |
|------|------|----------|
| **1. 表格视图** | 核心数据展示 | 列自定义、排序筛选、行列操作、分页 |
| **2. 看板视图** | 任务管理 | 按字段分组、卡片拖拽、多级泳道 |
| **3. 日历视图** | 时间管理 | 月/周/日/年视图、颜色编码、拖拽调整 |
| **4. 画廊视图** | 内容展示 | 网格/瀑布流布局、图片预览、封面提取 |
| **5. 列表视图** | 简化展示 | 紧凑布局、层级缩进、行内操作 |
| **6. 图表视图** | 数据可视化 | 柱状图/折线图/饼图/散点图/热力图 |
| **7. 图谱视图** | 关系可视化 | 力导向布局、社区发现、不同类型节点 |
| **8. 表单视图** | 数据录入 | 类型控件渲染、实时验证、多步骤向导 |

### 智能视图推荐

- 时间字段多 → 日历视图
- 有关联字段 → 图谱视图
- 有状态字段（status/stage）→ 看板视图
- 默认 → 表格视图

视图定义支持三种方式：Markdown 内联注释、YAML 配置文件、编程式 API。

## 五、Markdown vs CSV vs JSON — 格式对比

### 结构化程度

| | Markdown | CSV | JSON |
|--|----------|-----|------|
| **优点** | 富文本、可嵌套、人类友好、双向链接 | 结构规整、解析简单、工具生态丰富 | 结构灵活、类型明确、嵌套好 |
| **缺点** | 结构松散、解析复杂、无标准 Schema | 不支持嵌套、无类型、转义问题 | 冗余度高、人阅读难、无 schema |
| **适用** | 文档型数据、知识库 | 表格数据、数据交换 | API 数据、配置、复杂对象 |

### 性能对比

| 指标 | Markdown | CSV | JSON |
|------|----------|-----|------|
| 解析速度 | **慢** | 快 | 中 |
| 存储效率 | **低** | 高 | 中 |
| 查询性能 | **低** | 中 | 中 |
| 编辑友好 | **高** | 低 | 中 |
| Git 友好 | **行级 diff** | 整行变化 | 格式化敏感 |

### 功能特性

| 特性 | Markdown | CSV | JSON |
|------|----------|-----|------|
| 类型系统 | 文本、链接、格式、嵌入 | 字符串、数字（需推断） | 完整类型系统 |
| Schema 支持 | 弱（Frontmatter/YAML） | 无（仅表头） | 弱（JSON Schema 可选） |
| 关系支持 | Wiki 链接、双向链接 | 外键（需应用层维护） | 对象引用 |
| 版本控制 | Git 友好（行级 diff） | Git 不友好（整行变化） | Git 不友好（格式化敏感） |

## 六、核心挑战与缺点

### 四大核心挑战

| 类别 | 具体挑战 |
|------|----------|
| **结构化挑战** | 语法宽松、混合内容、无强 Schema、类型信息丢失 |
| **性能挑战** | 文件 I/O 瓶颈、Markdown 解析开销、索引维护成本高、内存消耗大 |
| **并发挑战** | 文件锁冲突、无事务隔离、Git 合并冲突人工解决、实时同步延迟 |
| **查询挑战** | 无原生 B+树/哈希索引、跨文件 JOIN 低效、聚合需全表扫描、子查询难优化 |

### 关键局限

- **数据规模**：不适合百万级+记录
- **ACID**：难以实现完整事务特性
- **工具链**：缺少 ORM、迁移工具、GUI 管理工具
- **高频协作**：无原生 RBAC、行级权限，并发写冲突需 CRDT/Git 解决

## 七、适用场景 & 不适用场景

### ✅ 适合 Markdown 数据库

- 个人知识管理（PKM）/ 第二大脑
- 文档型 CMS
- 研发文档系统 / Dev Journal / Incident Log
- 个人 CRM / 任务追踪
- 静态站点生成
- 文献引用与研究笔记数据库
- 案例/判例/客户档案（合规/隐私要求）

### ❌ 不适合

- 高并发 OLTP（电商/ERP）
- 百万级+数据量
- 强权限管控的多人协作
- 复杂外键/触发器/存储过程
- 实时多人共同编辑
- 金融核心系统 / 物联网时序数据

## 八、混合架构方案

| 方案 | 实现方式 | 优势 |
|------|----------|------|
| **分层存储** | SQLite 存索引和关系，Markdown 存正文 | 查询快 + 内容可读 |
| **缓存加速** | Markdown 为源，内存数据库（Redis）为缓存 | 实时查询 + 文件持久化 |
| **增量索引** | 文件监听 + 后台增量索引服务 | 响应快 + 资源占用低 |

## 十、总结

> **Markdown 数据库**最适合以文本为核心资产、需要结构化查询和版本追溯、重视数据主权和长期可移植性的个人或小团队场景。
>
> 关键是在**不改变 Markdown 核心文件格式**的前提下，在其上建立数据库的抽象层——用户看到的依然是普通的 Markdown 文件，但获得了数据库的强大能力。类似 Git 在文件系统上构建版本控制，我们在 Markdown 上构建知识数据库。

**典型用户画像**：Obsidian 重度用户 + 有一点技术背景 + 不想把重要数据锁在云端。
