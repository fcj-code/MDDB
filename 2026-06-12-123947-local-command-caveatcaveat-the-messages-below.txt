● 好，根因已经非常明确了。让我总结：

  Phase 1: 根因确认

  处理器被 Obsidian 在两个不同的渲染阶段重复调用：
  - e.sync — 初始 DOM 渲染
  - o.postProcess — 后处理器

  而且不仅是 2 次 — 每个处理器被调用了 6+ 
  次！两个阶段互相交叉，都在同一个同步执行流中。

  关键证据：
  1. mddb-form 的 sourceLen 值不同（93 vs 94）— 说明不同阶段传入的 source
  有微小差异
  2. 调用栈有两个不同的来源路径
  3. elChildren 始终为 0 — 说明 el.empty() 在每个调用都生效了

  根因： 同一个 el 在同一个事件循环中被 Obsidian 多次传入处理器，el.empty()
  每次都清空再渲染，导致最后虽然只有一个结果，但中间有多次不必要的查询和渲染，且
  用户看到的最终结果是最后一个调用的渲染 — 如果不同阶段使用不同的 
  el，则两个渲染结果共存。

  Phase 3: 修复方案

  用 queueMicrotask 做单次渲染保护：在同一个事件循环中，同一个 el 只渲染一次。

  修复总结

  根因（最终确认）

  通过你提供的日志，我确认了：

  Obsidian 在 Live Preview 
  模式下，在两个不同的渲染阶段调用了同一个代码块处理器：
  1. e.sync — 初始 DOM 渲染
  2. o.postProcess — Markdown 后处理器

  而且不是 2 次——每个处理器被调用了 6 次以上。mddb-form 的 sourceLen
  在两个阶段不同（93 vs 94），说明可能传入的是不同的 DOM 
  容器，导致两个渲染结果同时出现在页面中。

  修复方案

  从 el.empty() + generation 保护 改为 WeakSet 同一事件循环保护：

  const tableRendering = new WeakSet<HTMLElement>();
  this.registerMarkdownCodeBlockProcessor('mddb-table', (source, el) => {
    // 同一事件循环中已处理过此 el → 跳过（Obsidian 在 sync + postProcess 
  两个阶段都会调用）
    if (tableRendering.has(el)) return;
    tableRendering.add(el);
    queueMicrotask(() => tableRendering.delete(el));

    el.empty();
    // ... 渲染 ...
  });

  原理：WeakSet 标记当前事件循环中已处理过的 el，queueMicrotask
  在下一个微任务中清除标记。这样：
  - 同一事件循环中的重复调用 → 被跳过（只渲染一次）
  - 切换 tab 后的事件循环 → 标记已清除（正常重新渲染）
