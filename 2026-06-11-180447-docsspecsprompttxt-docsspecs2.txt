  ---
  MD-DB 实施路线图架构评审

  1. 整体架构合理性

  1.1 架构分层总体清晰，但“存储引擎 / 解析管道 / 查询引擎 / 
  事务模型”的边界仍有交叉

  风险等级：中

  当前路线图将系统拆为：

  - Obsidian 插件入口与设置层
  - 解析管道
  - SQLite / 绑定表 / 缓存
  - CRUD
  - WAL
  - 重扫与一致性
  - 查询引擎
  - 事务模型
  - 视图层

  整体分层是合理的，符合一个本地 Markdown
  数据库插件的复杂度。但存在一些职责边界交叉：

  1. Phase 9 CRUD 已包含文件写入、SQLite 更新、binding 更新。
  2. Phase 10 WAL 又描述“更新内存 SQLite → UI 立即响应 → 写文件失败生成 WAL”。
  3. Phase 16 事务模型 再次定义单文件 / 跨文件写入路径，并复用 WAL。
  4. Phase 11 重扫与一致性 也参与写前哈希验证、冲突检测。
  5. Phase 17 视图层 又包含客户端校验、CRUD、配置写回、跨视图同步。

  这会导致实现时容易出现多个模块都“认为自己负责写入一致性”的问题。

  改进建议：

  建议增加一节“写入路径责任边界”，明确：

  View Layer
    只负责收集用户意图、展示状态、触发 Engine API

  Engine / Transaction Layer
    统一负责所有写入入口、事务边界、冲突策略、WAL 生成

  CRUD Layer
    只负责最小原子操作：insert/update/delete 单条记录的文件与索引变更

  Storage / Binding Layer
    只负责 SQLite 表、binding、schema_registry、file_hashes 的读写

  Rescan Layer
    只负责外部变化检测和索引重建，不直接处理业务事务

  WAL Layer
    只负责失败补偿、重试、死信，不决定业务语义

  尤其需要明确：所有写入必须经过 TransactionManager，不应允许 View 或 CRUD 
  直接绕过事务层写文件。

  ---
  1.2 架构风格与业务规模基本匹配

  风险等级：低

  MD-DB 是 Obsidian
  插件内运行的本地数据库引擎，不适合微服务或外部数据库架构。当前采用：

  - 单进程插件
  - 本地 WASM SQLite
  - Markdown 文件作为持久化源
  - 内存索引 + 文件写穿
  - JSON WAL 补偿

  这个架构风格与 Obsidian 插件环境匹配，复杂度总体可控。

  但它已经接近一个“小型嵌入式数据库引擎”，设计复杂度明显高于普通 Obsidian
  插件。路线图中 Phase 0-17 都标记为“可直接实施”，实际风险偏乐观。

  改进建议：

  建议将实施状态拆为：

  - ✅ 规格充分，可进入实现
  - ⚠️ 需要实现前补充接口契约
  - ⬜ 需要重新设计

  例如 Phase 10、11、12、16
  不应仅标为“可直接实施”，它们需要更精确的状态机、失败恢复流程和接口契约。

  ---
  1.3 当前路线图更像“模块清单”，还缺少系统级运行时架构说明

  风险等级：中

  路线图列出了很多 Phase，但没有完整描述系统启动后各组件如何组合，例如：

  - Engine 是否是唯一顶层 facade？
  - ParsePipeline、QueryEngine、TransactionManager、WalManager
  的依赖注入关系是什么？
  - SQLite 实例由谁创建、谁持有、谁释放？
  - file lock、schema registry、binding store 是否是单例？
  - 插件卸载时如何 flush / dispose / stop retry loop？

  改进建议：

  建议补充一张“运行时组件图”，例如：

  PluginMain
   └── MDDBEngine
        ├── StorageManager
        │    ├── SQLiteAdapter
        │    ├── BindingStore
        │    ├── SchemaRegistry
        │    └── FileHashStore
        ├── ParsePipeline
        │    ├── SchemaResolver
        │    ├── Lexer
        │    ├── TypeConverter
        │    ├── Validator
        │    └── IndexWriter
        ├── TransactionManager
        │    ├── CrudExecutor
        │    ├── ConflictDetector
        │    └── WalManager
        ├── QueryEngine
        ├── RescanScheduler
        ├── FileWatcher
        └── ViewLayer

  并明确所有模块的生命周期。

  2. 可扩展性与演进性

  2.1 多文件表与 A/C 形式仍未设计，但前置架构已大量假设单文件表

  风险等级：高

  路线图中 Phase 18 多文件表、Phase 19 A/C
  适配层标记为“需要先设计”。但前面的核心设计已经深入绑定了当前 B
  形式、单文件表模型，例如：

  - schema_registry.json 使用 table -> info，其中 file: string
  - _binding 使用 table_name + logical_pk 唯一索引
  - CRUD INSERT 默认为“追加到文件末尾”
  - 查询引擎假设单表主要来自单个 SQLite 用户表
  - 事务路径通过“涉及文件数”判断单文件 / 跨文件
  - Phase 13 delete 文件时有 DROP TABLE IF EXISTS [table]

  这些设计对多文件表会产生明显冲突。

  例如一张表跨多个文件时：

  1. schema_registry.tables[table].file 不能再是单一字符串。
  2. 删除某个文件不能直接 DROP TABLE，只能删除该文件贡献的行。
  3. INSERT 需要知道插入到哪个分区文件。
  4. _binding(logical_pk, table_name) 的唯一性是否跨分区全局生效需要明确。
  5. 查询时是否需要分区裁剪、是否所有分区都映射到同一个 SQLite 表，也需要明确。

  具体改进建议：

  在 Phase 2 之前就预留多文件表模型，至少将 schema registry 结构调整为：

  interface SchemaRegistry {
    version: 1;
    tables: Record<string, {
      table: string;
      schema: SchemaSummary;
      sources: Array<{
        file: string;
        blockId?: string;
        rowCount: number;
        partition?: Record<string, string>;
      }>;
    }>;
  }

  并将 Phase 13 的 delete 行为改为：

  删除文件时：
  1. 删除 _binding 中 file_path = ? 的记录
  2. 删除用户表中属于该 file_path 的记录
  3. 如果该表没有任何 source，再 DROP TABLE
  4. 更新 schema_registry.tables[table].sources

  同时建议将 Phase 18 提前为架构前置设计，不应等 Phase 15 之后才设计。

  ---
  2.2 Storage PK 依赖物理行号和短 hash，长期演进风险较高

  风险等级：高

  当前 Storage PK 格式：

  `${relativePath}:${lineNumber}:${sha256(rawContent).slice(0, 6)}`

  问题包括：

  1. 行号会漂移：虽然 Phase 11 设计了后续行号更新，但用户外部编辑、批量插入、格
  式化、合并冲突后，行号可能大规模变化。
  2. hash 只有 6 位十六进制：约 24 bit，数据量增加后碰撞概率不可忽略。
  3. storage_pk 不可变 与 lineNumber 可变天然冲突。Phase 11 说 UPDATE 第 N 行时
  storage_pk 不变，但如果 DELETE 前面的行导致 line number 变化，则 storage_pk
  中嵌入的 lineNumber 已过期。
  4. 如果 storage_pk 是外部 API 暴露给视图层的记录 ID，那么行号变化会让 ID
  语义混乱。

  具体改进建议：

  建议重新定义：

  storage_pk = stableRowId

  优先方案：

  file_path + block_id + row_id

  其中：

  - block_id：mddb 代码块的稳定 ID，可由块起始位置 + schema + 内容 hash
  初始化，之后维护。
  - row_id：如果用户数据中无显式 ID，可隐藏生成 _mddb_id，或者在 binding
  中保存首次发现时生成的 UUID。
  - line_number 仅作为 binding 元数据，不应进入主键。

  如果不希望修改 Markdown 原文加入隐藏 ID，则至少改为：

  interface BindingRow {
    storage_pk: string;     // UUID or long hash
    file_path: string;
    block_index: number;
    line_number: number;
    row_hash: string;       // 建议 128 bit 以上
    logical_pk: string;
  }

  并把 line_number 视为可更新定位信息，而不是身份的一部分。

  2.3 查询、视图、事务均已设计完成，但缺少插件 API 的统一 facade

  风险等级：中

  路线图中有 QueryEngine.query()、engine.insert/update/delete、engine.transactio
  n(cb)、ViewModel CRUD 等多个入口，但没有定义一个统一的 public API。

  这会影响后续扩展：

  - 插件内部视图如何调用？
  - 第三方插件是否可调用？
  - 命令面板如何触发？
  - 测试如何 mock？
  - 未来 A/C adapter 如何接入？

  改进建议：

  建议补充 MDDBEngine 顶层接口：

  interface MDDBEngine {
    initialize(): Promise<void>;
    shutdown(): Promise<void>;

    parseFile(file: TFile): Promise<ParseResult>;
    rescanVault(): Promise<RescanResult>;

    query(q: Query): ResultOrError<ResultSet>;
    queryRaw(sql: string, params?: unknown[]): ResultOrError<ResultSet>;

    insert(table: string, record: RecordInput, options?: WriteOptions):
  Promise<WriteResult>;
    update(storagePk: string, patch: RecordPatch, options?: WriteOptions):
  Promise<WriteResult>;
    delete(storagePk: string, options?: WriteOptions): Promise<WriteResult>;

    transaction<T>(cb: (tx: Transaction) => Promise<T>): Promise<T>;

    on(event: EngineEventType, cb: EngineEventHandler): Disposable;
  }

  所有上层模块只依赖这个 facade。

  3. 高可用与容错设计

  3.1 WAL 设计方向正确，但当前 WAL 内容不足以完成可靠重放

  风险等级：高

  Phase 10 的 WAL 格式目前只记录：

  interface WalEntry {
    txId: string;
    createdAt: string;
    status: "pending" | "retrying" | "dead";
    files: Array<{
      relativePath: string;
      tableName: string;
    }>;
    retry: ...
  }

  这个格式不足以在写失败或重启后完成可靠重放，因为它没有记录：

  - 操作类型：insert / update / delete / batch / transaction
  - 写入前内容
  - 写入后内容
  - 目标行定位信息
  - row_hash / expected_hash
  - SQLite 已提交状态
  - 文件写入阶段进度
  - 多文件事务中每个文件是否已写成功
  - 重试是否幂等
  - 如果用户在重试前手动修改了文件，该如何处理

  尤其 Phase 16 说跨文件事务路径是：

  SQLite COMMIT → 写 WAL → 逐个 vault.process()

  如果第 1 个文件写成功、第 2 个文件写失败，当前 WAL
  只知道“涉及哪些文件”，不知道每个文件要执行什么变更，也不知道哪些文件已成功。

  这是非常关键的崩溃恢复缺口。

  具体改进建议：

  WAL 应记录可重放操作，而不是只记录受影响文件。建议改为：

  interface WalEntry {
    txId: string;
    version: 1;
    createdAt: string;
    status: "pending" | "retrying" | "dead" | "done";

    operations: WalOperation[];

    progress: {
      completedOperationIds: string[];
    };

    retry: {
      count: number;
      maxRetries: number;
      lastError: string | null;
      lastAttemptAt: string | null;
      nextAttemptAt: string | null;
    };
  }

  type WalOperation =
    | {
        id: string;
        type: "insertLine";
        file: string;
        afterLine?: number;
        content: string;
        expectedFileHash?: string;
      }
    | {
        id: string;
        type: "replaceLine";
        file: string;
        lineNumber: number;
        beforeHash: string;
        beforeContent?: string;
        afterContent: string;
      }
    | {
        id: string;
        type: "deleteLine";
        file: string;
        lineNumber: number;
        beforeHash: string;
        beforeContent: string;
      };

  同时定义重放规则：

  1. 如果 operationId 已在 completedOperationIds 中，跳过。
  2. 执行前检查 beforeHash / fileHash。
  3. 如果目标内容已经等于 afterContent，视为幂等成功。
  4. 如果目标内容既不是 beforeContent 也不是 afterContent，进入 conflict/dead。
  5. 每完成一个 operation，立即持久化 progress。
  6. 全部完成后删除 WAL 或标记 done 后延迟清理。

  这是高风险项，建议在正式实施 Phase 10 / Phase 16 前必须补齐。

  ---
  3.2 “内存 SQLite 先更新、文件后写”的一致性语义不够严谨

  风险等级：高

  Phase 10 写入路径：

  1. 更新内存 SQLite → UI 立即响应
  2. 调用 app.vault.modify() 写文件
  3. 成功 → 流程结束
  4. 失败 → 生成 wal/{txId}.json → 后台指数退避重试

  这个设计是乐观 UI，但如果文件写失败、WAL 又写失败，或者插件进程崩溃发生在
  SQLite 更新之后、WAL 持久化之前，会出现：

  - UI 看到成功
  - SQLite 内存中有数据
  - Markdown 文件没有落盘
  - WAL 也没有记录
  - 重启后数据丢失或状态倒退

  Phase 16 又说跨文件事务：

  SQLite COMMIT → 写 WAL → 逐个 vault.process()

  这比 Phase 10 更安全，但仍需要明确顺序。

  具体改进建议：

  对于所有非纯查询写入，建议采用：

  1. 构造 WritePlan
  2. 持久化 WAL，状态 pending
  3. 应用 SQLite 临时事务
  4. 写 Markdown 文件
  5. 文件写成功后提交 SQLite / 更新 binding / file_hashes
  6. 标记 WAL done / 删除 WAL

  如果为了 UI 响应必须乐观更新，则要显式引入状态：

  type SyncState = "synced" | "pending" | "retrying" | "dead";

  并让视图层展示“未同步”状态，而不是把失败写入伪装成成功。

  关键原则：

  文件是事实源时，不能让 SQLite 成为唯一已提交事实。
  WAL 必须先于不可回滚的内存状态持久化。

  ---
  3.3 跨文件事务“最终一致，向前推进”可以接受，但需明确用户可见语义

  风险等级：中

  Phase 16 明确：

  单文件完美回滚
  跨文件最终一致性，不可回滚，向前推进

  这个选择在 Obsidian Vault API 环境下是现实的。但路线图没有说明：

  - 跨文件事务部分成功时，用户看到什么？
  - 查询结果是否包含已提交但未写完的变更？
  - dead WAL 是否会阻塞后续写入？
  - 用户点击“丢弃”时，是丢弃 WAL 还是回滚 SQLite？
  - 如果文件被用户修改导致 WAL 无法继续向前推进，如何解决？

  改进建议：

  补充跨文件事务状态机：

  preparing → wal_written → applying_files → retrying → synced
                                        └── dead

  并定义用户可见策略：

  pending/retrying:
    查询默认读取 SQLite 当前状态，但 ResultSet metadata 标记 hasPendingWrites =
  true。

  dead:
    状态栏强提示。
    涉及表可继续读，但相关记录标记 syncState = dead。
    后续写入同一文件前必须提示用户处理死信。

  ---
  3.4 文件锁设计过于简单，可能无法覆盖异步重入和插件生命周期

  风险等级：中

  Phase 12 使用：

  lockedFiles: Set<string>

  作为全局锁。这个方案可以做轻量互斥，但需要注意：

  - vault.process() 是异步的，必须保证 finally unlock。
  - 多个操作可能锁多个文件，存在死锁风险。
  - rename/delete/modify 事件可能在写入过程中触发。
  - 同一个事务中多文件锁顺序必须稳定。
  - 插件卸载时正在持锁怎么办？

  改进建议：

  补充 LockManager：

  interface LockManager {
    withFileLock<T>(path: string, fn: () => Promise<T>): Promise<T>;
    withFileLocks<T>(paths: string[], fn: () => Promise<T>): Promise<T>;
  }

  要求：

  1. 多文件锁按路径排序获取，避免死锁。
  2. 所有锁通过 finally 释放。
  3. 支持 reentrant ownerId，允许同一事务重复进入。
  4. 插件卸载时拒绝新锁，等待或取消已有任务。

  ---
  4. 性能与容量设计

  4.1 缺少明确容量目标和性能预算

  风险等级：高

  路线图中有 <100ms 冷启动目标，但缺少系统级容量假设，例如：

  - 支持多少 Markdown 文件？
  - 支持多少 mddb 块？
  - 单表多少行？
  - 总记录数多少？
  - 单文件最大行数？
  - 查询目标延迟是多少？
  - 冷启动缓存多大？
  - WAL 最大积压多少？
  - 日志文件增长上限？

  没有容量目标，很多设计无法判断是否合理。例如：

  - 逐行 diff 是否能承受 10 万行文件？
  - 全量 parseAllFiles 是否能承受 1 万个 Markdown 文件？
  - sql.js WASM 内存是否足够？
  - ResultSet 是否允许一次返回全部行？
  - 虚拟滚动是否只是 DOM 虚拟，还是查询分页也虚拟？

  具体改进建议：

  在路线图前部增加“容量与性能目标”：

  MVP 目标：
  - Vault 文件数：≤ 5,000 markdown files
  - mddb 数据文件：≤ 500 files
  - 总记录数：≤ 100,000 rows
  - 单表记录数：≤ 50,000 rows
  - 单文件数据行：≤ 5,000 rows
  - 冷启动可交互：≤ 300ms
  - 后台全量校验：≤ 30s，分批执行，不阻塞 UI
  - 常规查询 P95：≤ 50ms
  - CRUD 单条写入 P95：≤ 100ms
  - WAL retry 不阻塞 UI

  并区分：

  MVP 支持规模
  推荐规模
  超过规模后的降级行为

  ---
  4.2 parseAllFiles() 全 Vault 扫描存在性能热点

  风险等级：中

  Phase 8：

  parseAllFiles()
  → 遍历 vault.getMarkdownFiles()，过滤含 mddb 块或 @table frontmatter 的文件

  如果每次全量扫描都读取所有 Markdown 内容，在大型 Obsidian Vault 中会明显卡顿。

  改进建议：

  引入文件候选索引：

  1. 首次扫描建立 mddb_files.json。
  2. 后续启动优先读取 mddb_files.json。
  3. create/modify 事件增量更新候选集。
  4. 全量扫描只在用户手动 Rescan 或缓存损坏时执行。

  并且扫描应分批：

  interface ScanOptions {
    batchSize: number;       // e.g. 20 files
    yieldEveryMs: number;    // e.g. 16ms
    signal?: AbortSignal;
  }

  ---
  4.3 增量 diff 方案需要更具体，否则大文件会成为热点

  风险等级：中

  Phase 11：

  逐行 diff → ≤20% 增量 / >20% 全量

  这个策略方向合理，但缺少：

  - diff 算法选择
  - 是否保存上一版本完整文件内容
  - 大文件内存成本
  - 行移动检测策略
  - 多个 mddb 块时如何定位变更属于哪个表
  - 如果用户插入一行导致后续行号全部变化，如何避免误判大面积变更

  改进建议：

  建议明确：

  MVP 使用简单 line hash diff：
  - 保存每个数据文件的 lineHashes。
  - modify 时重新计算当前 lineHashes。
  - 使用前后缀快速匹配检测插入/删除。
  - 超过阈值时整文件重建该文件贡献的索引。

  并把“逐行 diff”封装成独立模块，避免散落在 FileWatcher 或 RescanScheduler 中。

  ---
  4.4 sql.js 持久化策略不清晰

  风险等级：中

  Phase 2 说：

  binding.db # 绑定表 + 索引 SQLite

  Phase 12 说冷启动加载 binding.db。

  但 sql.js 是 WASM SQLite，通常需要显式 export database binary
  再写入文件。路线图没有说明：

  - 何时把内存 SQLite 导出到 binding.db
  - 每次写入都 export 是否过重
  - 崩溃前未 export 的数据如何恢复
  - 用户表是否也在同一个 SQLite 文件中持久化
  - binding.db 是否包括用户表数据，还是只包括 binding

  改进建议：

  明确 SQLite 持久化策略：

  - SQLite 内存库包含：_binding + 用户表。
  - binding.db 是整个 SQLite database 的二进制快照。
  - 每次事务成功后不立即全量 export，而是 debounce 1-5s 持久化。
  - 插件 unload 时强制 flush。
  - 如果 binding.db 落后于 Markdown 文件，以 file_hashes 校验后后台修复。
  - WAL 重放优先于后台验证。

  或者把命名从 binding.db 改成更准确的：

  mddb-cache.sqlite

  避免误以为只存 binding。

  5. 安全性与合规性

  5.1 raw SQL 是必要 escape hatch，但需要更严格的安全边界

  风险等级：高

  Phase 15 支持：

  queryRaw(sql, params)

  并说明使用 prepared statement 防注入。

  但在本地插件环境中，raw SQL 仍然有风险：

  - 用户可能执行 DROP TABLE
  - 视图代码块如果允许 raw SQL，会让 Markdown 文件具备破坏缓存能力
  - 第三方同步来的 Markdown 可能包含恶意查询块
  - raw SQL 可能读取系统表 _binding
  - raw SQL 可能造成性能问题，例如大查询、笛卡尔积、递归 CTE

  即使不是传统 Web 安全问题，也属于本地数据完整性和插件稳定性风险。

  具体改进建议：

  明确 raw SQL 分层权限：

  1. 内部 API queryRawInternal：允许完整 SQL，仅插件内部使用。
  2. 用户 API queryRaw：默认只允许 SELECT。
  3. 视图层声明式代码块：禁止 raw SQL，除非用户在设置中显式开启高级模式。
  4. 禁止多语句。
  5. 禁止 DDL / DML：DROP, DELETE, UPDATE, INSERT, ALTER, PRAGMA 等。
  6. 限制执行时间和返回行数。

  建议接口：

  queryRaw(sql: string, params?: unknown[], options?: {
    readonly?: boolean;      // default true
    maxRows?: number;
    timeoutMs?: number;
  })

  ---
  5.2 表名、字段名用于 SQL 生成，必须定义 identifier escaping

  风险等级：高

  Schema 来自 Markdown 内容：

  @table
  @fields
  @sort
  @indexes

  这些值最终会生成 SQLite DDL / SQL。如果没有严格校验和转义，可能产生 SQL 注入或
  SQL 语法破坏。

  例如恶意字段名：

  name); DROP TABLE _binding; --

  虽然 sql.js 本地执行，但会破坏插件缓存和用户体验。

  具体改进建议：

  在 Phase 3 Schema 元验证中增加：

  table name:
    /^[A-Za-z_][A-Za-z0-9_]*$/

  field name:
    /^[A-Za-z_][A-Za-z0-9_]*$/

  或者如果要支持中文字段名，则必须统一 quote identifier：

  function quoteIdent(name: string): string {
    return `"${name.replace(/"/g, '""')}"`;
  }

  并规定：

  1. 所有表名、列名、索引名必须通过 quoteIdent。
  2. 所有值必须通过 prepared statement 参数绑定。
  3. @sort、@indexes 不得直接拼接原始字符串，必须 parse 成 AST 后生成。

  这是高风险项，应作为 Phase 3 / Phase 7 / Phase 15 的硬性要求。

  5.3 敏感数据、日志脱敏与隐私策略不足

  风险等级：中

  MD-DB 可能存储财务、联系人、电话、邮箱等数据。路线图中有 phone、email
  类型，也有日志系统：

  .logs/{YYYY-MM-DD}.log
  [HH:MM:SS] [LEVEL] [TABLE] message

  但没有说明：

  - 日志是否记录原始行内容
  - 错误是否包含字段值
  - email/phone 是否脱敏
  - WAL 是否可能存储用户敏感数据
  - cache / WAL / logs 是否清理
  - 用户是否可以关闭日志

  改进建议：

  增加安全策略：

  1. 默认日志不记录完整记录内容，只记录 table、file、line、errorCode。
  2. phone/email/text 字段在日志中默认脱敏。
  3. WAL 必然包含待写内容，设置中提示其可能包含敏感数据。
  4. 提供 Clear cache / Clear WAL / Clear logs 命令。
  5. 日志级别默认 warn，debug 需用户手动开启。

  示例脱敏：

  maskEmail("alice@example.com") => "a***@example.com"
  maskPhone("13812345678") => "138****5678"

  7. 技术选型与复杂度

  7.1 sql.js 作为优先方案合理，但需要明确限制

  风险等级：中

  选择 sql.js 的优点：

  - 纯 WASM
  - 无原生依赖
  - 更适合 Obsidian 插件分发
  - 跨平台更简单

  但需要注意：

  - WASM 加载时间
  - 内存占用
  - 数据库持久化需要手动 export
  - 大库写入性能可能不如原生 SQLite
  - Obsidian 移动端兼容性需要验证

  改进建议：

  在技术选型处补充：

  MVP 使用 sql.js。
  如果满足任一条件，评估迁移到 Obsidian 原生 SQLite 或其他后端：
  - 总记录数超过 500k
  - binding.db 超过 100MB
  - 冷启动加载超过 1s
  - 查询 P95 超过 200ms
  - 移动端内存不可接受

  并定义 StorageAdapter：

  interface SQLiteAdapter {
    exec(sql: string, params?: unknown[]): QueryRows;
    transaction<T>(fn: () => T): T;
    export?(): Uint8Array;
    close(): void;
  }

  避免 sql.js 侵入全系统。

  总体评价

  2026-06-10-implementation-roadmap.md 是一份质量较高的实施路线图。它的优点是阶
  段拆分清楚、模块边界初步明确、依赖关系完整，并且能让子代理或其他 session 按
  Phase
  独立推进。作为“从设计文档转为工程任务”的中间产物，它已经具备很强的可执行性。

  1. WAL 与事务恢复语义不足，这是最高风险点。
