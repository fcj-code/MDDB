● 好，现在深挖解析管道。这是存储引擎的第一个入口——把 .md
  原始文本变成类型化的结构化记录。我先梳理已有基础，然后逐层展开讨论。

  ---
  解析管道：整体架构

  回顾已有的伪代码框架（来自 markdown数据库.md）：

  文件内容 → 提取元数据块 → 解析字段定义 → 逐行切分 → 类型转换 → 结构化记录

  展开为完整的解析管道，应该分为 6 个阶段：

  Phase 0: 文件发现        → 遍历 Vault，识别含 @table 注释的 .md 文件
  Phase 1: Schema 提取     → 解析 HTML 注释块，构建 Schema 对象
  Phase 2: 行词法分析      → 逐行扫描，识别数据行/注释行/空行，按 | 切分字段
  Phase 3: 类型转换        → 每个字段值按 @types 声明转为强类型
  Phase 4: 验证            → 类型校验 + 必填检查 + 枚举约束 + 唯一性
  Phase 5: 写入索引        → 生成存储 PK → 插入绑定表 → 写入内存 SQLite

  Phase 0-1 在冷启动/文件变更时触发，Phase 2-5 逐行执行。

  ---
  Phase 0: 文件发现

  这部分相对简单，但有几个点需要定：

  - 全局扫描 *.md → 读文件头 4KB → 正则匹配 @table
  - schema_registry.json 缓存已发现的表文件路径和最后修改时间
  - 文件重命名 → vault.on('rename') → 更新 _binding.file_path

  一个决策点：一个 .md 文件允许定义多张表吗？

  ┌───────────────────────────────┬──────────────────────────────────────┐
  │             选项              │                 说明                 │
  ├───────────────────────────────┼──────────────────────────────────────┤
  │ A: 一个文件 = 一张表          │ 简单，文件路径即表标识，当前设计假设 │
  ├───────────────────────────────┼──────────────────────────────────────┤
  │ B: 一个文件可含多个 @table 块 │ 灵活但复杂，需要块级寻址             │
  └───────────────────────────────┴──────────────────────────────────────┘

  建议 A，保持简单。如果一个文件可以有多张表，解析器复杂度翻倍且边界情况多（行属
  于哪个表？块标记怎么放？）。

  ---
  Phase 1: Schema 提取

  这是解析管道的"配置加载"。从 HTML 注释块中提取指令：

  <!--
  @table transactions
  @pk (日期, 金额, 商户)
  @sort (日期 ASC)
  @fields 日期 | 金额 | 类型 | 分类 | 账户 | 商户 | 备注
  @types  date | decimal(2) | enum(支出,收入) | string | ref(accounts) | string
  | text
  @required true | true | true | true | true | true | false
  @indexes idx(分类) | idx(账户) | idx(金额)
  -->

  解析规则：
  - 以 <!-- 开始、--> 结束的 HTML 注释块
  - 文件首个这样的块为 Schema（也允许 @schema_start / @schema_end 显式标记）
  - 每行一个指令：@directive_name [value]
  - | 分隔多值指令（@fields, @types, @required,
  @indexes）——值的数量必须精确匹配字段数

  Schema 验证（元验证——验证 Schema 本身是否合法）：
  - @table 和 @pk 和 @fields 和 @types 为必需指令
  - @fields 和 @types 的列数必须相等
  - @required 如果存在，列数也必须相等
  - @pk 中的列名必须在 @fields 中存在
  - @sort 中的列名必须在 @fields 中存在

  ---
  Phase 2: 行词法分析 — 核心难点

  这是解析管道最复杂的阶段。原始文本行 → 字段值数组。

  2.1 数据行识别

  文件中的每一行需要分类。分类规则：

  ┌───────────────┬───────────────────────────────────────┬──────────────┐
  │    行类型     │               识别规则                │     动作     │
  ├───────────────┼───────────────────────────────────────┼──────────────┤
  │ Schema 块     │ 匹配 <!-- / -->                       │ Phase 1      │
  ├───────────────┼───────────────────────────────────────┼──────────────┤
  │ 空行          │ 仅含空白字符                          │ 跳过         │
  ├───────────────┼───────────────────────────────────────┼──────────────┤
  │ Markdown 注释 │ 以 #、>、-  等开头？                  │ 待定         │
  ├───────────────┼───────────────────────────────────────┼──────────────┤
  │ 数据行        │ 含至少一个 | 且不以 Markdown 语法开头 │ 进入 Phase 2 │
  └───────────────┴───────────────────────────────────────┴──────────────┘

  关键决策：数据行和 Markdown 正文如何区分？

  方案 A：文件只允许 Schema 注释块 + 数据行，不能混入其他 Markdown
  内容（严格模式）

  方案 B：允许在数据行之间混入普通 Markdown 段落，引擎跳过非数据行（宽松模式）

  <!-- @table ... -->
  2024-06-01 | -45.00 | 支出 | 餐饮 | ...

  这里可以写一段备注文字吗？
  这是对这个月消费的总结。

  2024-06-02 | -128.00 | 支出 | 餐饮 | ...

  我倾向方案 A（严格模式）。理由：
  - 数据文件 = 表，不需要混入自由文本
  - 如果想写备注，应该用专门的 备注 字段
  - 宽松模式意味着引擎永远无法确定一行是不是数据，误判风险高
  - 但如果用户确实想在文件中写自由文本呢？——可以用 <!-- 备注 --> 注释块、或者以
  // 开头的行注释

  2.2 字段切分

  原始行: "2024-06-01 | -45.00 | 支出 | 餐饮 | 现金 | 兰州拉面 | 加了一份牛肉"

  朴素切分: line.split('|') → 7 个值

  核心问题：分隔符冲突

  text 类型字段和 string 类型字段的值可能包含 | 字符。例如：

  2024-06-01 | -45.00 | 支出 | 餐饮 | 现金 | 兰州拉面 |
  老板说"今天牛肉卖完了|明天补"

  这里 | 出现在备注文本中，朴素 split 会切成 8 个字段，导致整行错位。

  解决方案：

  ┌─────────────┬───────────────────────────────┬───────────────────────────┐
  │    方案     │             描述              │          优缺点           │
  ├─────────────┼───────────────────────────────┼───────────────────────────┤
  │ A:          │ | 表示字面量管道符            │ 标准做法，需引擎处理转义  │
  │ 反斜杠转义  │                               │                           │
  ├─────────────┼───────────────────────────────┼───────────────────────────┤
  │ B: text     │ 最后一个字段之后不可能再有分  │ 简单但有约束              │
  │ 必须放最后  │ 隔符                          │                           │
  ├─────────────┼───────────────────────────────┼───────────────────────────┤
  │ C: 引号包裹 │ "值包含|管道"                 │ 可读性好但引号本身也需要  │
  │             │                               │ 转义                      │
  ├─────────────┼───────────────────────────────┼───────────────────────────┤
  │ D: 双管道分 │ || 作为字段分隔               │ 降低冲突概率但不能根治    │
  │ 隔符        │                               │                           │
  └─────────────┴───────────────────────────────┴───────────────────────────┘

  我建议 A + B 组合：
  - \| 为转义的管道符，引擎在 split 前先扫描转义
  - 同时建议所有 text 类型字段放在最后（作为最佳实践而非强制约束）
  - 转义规则：\\ → 字面量反斜杠，\| → 字面量管道符，\n → 换行符（仅 text 类型）

  2.3 空白处理

  values = [v.strip() for v in line.split('|')] — 这个朴素做法有问题吗？

  ┌────────────────────────┬────────────┬─────────────┐
  │          场景          │ strip 行为 │  是否正确   │
  ├────────────────────────┼────────────┼─────────────┤
  │ " 兰州拉面 "           │ "兰州拉面" │     ✅      │
  ├────────────────────────┼────────────┼─────────────┤
  │ "" (空字段)            │ ""         │ ✅→映射为 - │
  ├────────────────────────┼────────────┼─────────────┤
  │ " " (仅空格)           │ ""         │ ✅→映射为 - │
  ├────────────────────────┼────────────┼─────────────┤
  │ " 一个 值 " (内部空格) │ "一个 值"  │     ✅      │
  ├────────────────────────┼────────────┼─────────────┤
  │ "   " (全空白)         │ ""         │ ✅→映射为 - │
  └────────────────────────┴────────────┴─────────────┘

  strip()
  是正确的默认行为。字段值的语义空白（如代码缩进）极少出现在这种表格数据中。

  2.4 空值识别

  在类型转换之前，先做空值检测：

  if trimmed == '' or trimmed == '-':
      value = NULL_SENTINEL  # 传给 Phase 3 的特殊标记

  @null_marker 自定义时，用配置的占位符字符串替换 -。

  ---
  Phase 3: 类型转换 — 逐类型深挖

  这是解析管道的核心。每个字段值从原始字符串转换为强类型值。

  3.1 类型转换器表

  string

  转换: 原样保留（去除首尾空白后）
  失败: 不会失败
  示例: "兰州拉面" → "兰州拉面"

  integer

  转换: parseInt(value.trim())
  合法格式: /^-?\d+$/
  失败: 非法字符 ("42abc")、浮点形式 ("42.0")、空值(NULL_SENTINEL) → NULL
  示例: "42" → 42, "-17" → -17

  decimal(N)

  转换: parseFloat(value.trim())，保留 N 位小数精度
  合法格式: /^-?\d+(\.\d{1,N})?$/
  内部存储: 整数形式（乘以 10^N），避免浮点精度问题
  示例: decimal(2): "45.00" → 4500 (内部)
         decimal(2): "45" → 4500 (补全小数位)
  显示: 始终格式化为 N 位小数

  decimal 的精度选择：decimal(2) 内部存为 BIGINT（分），避免 JS 浮点 0.1 + 0.2 =
  0.30000000000000004 问题。这是记账场景的核心需求。

  boolean

  合法真值: "true", "yes", "1", "是"
  合法假值: "false", "no", "0", "否"
  大小写不敏感
  其他: → NULL 或 报错（取决于严格模式）

  date

  转换: 解析为 ISO 日期
  合法格式:
    - YYYY-MM-DD (标准)
    - YYYY/MM/DD (兼容)
    - YYYY.MM.DD (兼容)
    - YYYY年MM月DD日 (中文习惯—待定)
  内部存储: ISO 字符串 "YYYY-MM-DD"
  范围校验: 年份 1900-2100，月份 1-12，日期对应该月有效
  示例: "2024-06-01" → "2024-06-01"
  非法: "2024-13-01", "昨天", "June 1 2024" → NULL

  datetime

  转换: 解析为 ISO datetime
  合法格式:
    - YYYY-MM-DD HH:MM:SS
    - YYYY-MM-DDTHH:MM:SS (ISO 8601)
    - YYYY-MM-DD HH:MM (秒默认 00)
  内部存储: ISO 字符串 "YYYY-MM-DD HH:MM:SS"
  示例: "2024-06-01 14:30:00" → "2024-06-01 14:30:00"

  enum(v1, v2, ...)

  转换: 精确匹配枚举值列表
  大小写: 敏感（"支出" ≠ "支出 "—trim 已处理尾部空白）
  匹配失败: → NULL (lenient) 或 报错 (strict)
  示例: enum(支出,收入): "支出" → "支出", "转账" → NULL

  text

  转换: 原样保留（支持任意字符，包括标点、空格）
  特殊处理: 转义还原（\| → |, \\ → \, \n → 换行）
  与 string 的区别: text 允许内部换行（转为 \n）、长度无上限

  tags

  转换: 从字符串提取 #tag 模式
  输入: "#技术 #重要 #云"
  输出: ["技术", "重要", "云"]
  分隔符: 空格 或 逗号
  标签内的空格: 不被支持（单个标签不含空格）
  去重: 保留顺序去重
  空输入: → 空数组 []

  ref(table)

  转换: 原样保留（字符串），标记为外键引用类型
  验证: Phase 4 检查目标表是否存在该值
  延迟验证: 冷启动时先跳过，后台验证补检查（目标表可能尚未加载）

  phone

  转换: 标准化为数字串
  合法格式:
    - 13812345678 (11位)
    - 138-1234-5678 (带分隔符)
    - +86 13812345678 (带国家码)
  内部存储: 纯数字串 "13812345678"
  失败: 非数字字符过多 → NULL

  email

  转换: 原样保留，trim + 小写
  验证: 正则 /^[^\s@]+@[^\s@]+\.[^\s@]+$/ 检查
  失败: → NULL

  3.2 类型转换失败处理

  这是一个重要的架构决策：

  ┌───────────────┬────────────────────────────────────┬────────────────────┐
  │     策略      │                行为                │      适用场景      │
  ├───────────────┼────────────────────────────────────┼────────────────────┤
  │ 宽松模式      │ 转换失败 → 存                      │ 默认模式，容错优先 │
  │ (lenient)     │ NULL，记录警告，继续解析           │                    │
  ├───────────────┼────────────────────────────────────┼────────────────────┤
  │ 严格模式      │ 转换失败 → 跳过整行，记录错误      │ 数据质量要求高     │
  │ (strict)      │                                    │                    │
  ├───────────────┼────────────────────────────────────┼────────────────────┤
  │ fallback 模式 │ 转换失败 →                         │ 调试/数据清洗      │
  │               │ 保留原始字符串，标记类型不匹配     │                    │
  └───────────────┴────────────────────────────────────┴────────────────────┘

  建议默认宽松模式。理由：
  - Markdown 数据库的核心价值是"人类可编辑"，人类会犯错
  - 一行数据里一个字段的日期写错不应导致整行被丢弃
  - 查询时 WHERE 金额 IS NULL 可快速定位脏数据
  - 未来可加 @strict true 指令切换严格模式

  ---
  Phase 4: 验证

  在类型转换之后、写入索引之前，进行约束验证。

  4.1 验证链

  类型验证 (Phase 3 已完成)
    → 必填检查
      → 枚举约束
        → 唯一性约束 (@pk)
          → 外键引用 (ref)

  4.2 必填检查

  if field.required and value is NULL:
      → 错误: "字段 '{field.name}' 为必填，但值为空"
      → 处理: 跳过该行 (strict) 或 存 NULL + 记录警告 (lenient, 默认)

  4.3 行宽校验

  切分出的字段数 vs Schema 定义的字段数：

  ┌──────────┬──────────────────────────────────────────────────────────────┐
  │   情况   │                             处理                             │
  ├──────────┼──────────────────────────────────────────────────────────────┤
  │ 相等     │ 正常                                                         │
  ├──────────┼──────────────────────────────────────────────────────────────┤
  │ 少于     │ 尾部字段填 NULL (lenient) 或 跳过 (strict)                   │
  │ Schema   │                                                              │
  ├──────────┼──────────────────────────────────────────────────────────────┤
  │ 多于     │ 尾部截断（因为最后一个字段通常是最宽松的 text 类型；默认     │
  │ Schema   │ lenient）或 跳过 (strict)                                    │
  └──────────┴──────────────────────────────────────────────────────────────┘

  这是一个容易被忽略的边界情况。用户手写时容易少写或多写字段。

  4.4 逻辑主键唯一性

  if @pk 值在 binding 表中已存在 (同表不同 storage_pk):
      → 错误: "逻辑主键冲突: (2024-06-01, -45.00, 兰州拉面)"
      → 处理: 跳过该行 (strict) 或 保留最新 (lenient, last-write-wins)

  4.5 外键引用完整性

  if field.type is ref(accounts):
      检查 accounts 表中是否存在该逻辑主键值
      检查时机: 写入时（目标表已加载）或 延迟到后台（目标表未加载）

  ---
  Phase 5: 写入索引

  验证通过的记录进入存储：

  1. 生成 storage_pk:
     format: "{rel_path}:{line_number}:{sha256(content)[:6]}"
