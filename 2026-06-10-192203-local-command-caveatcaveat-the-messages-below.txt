  我的初步分析

  在深入讨论之前，我想先抛出几个观察，作为讨论的起点：

  观察 A：单用户场景极大简化了并发控制。 不存在两个事务同时修改同一行的情况（用
  户不可能同时打两个键盘）。真正的"冲突"只有一个来源：引擎写入 vs 用户手动编辑
  vs Obsidian 自动保存。这意味着传统数据库的锁管理器可能完全是过度设计。

  观察 B：文件写入和内存写入是两个独立步骤。 当前设计是"先改内存 →
  再写文件"。如果在内存 SQLite 中开启事务、执行多条
  SQL、提交事务——但文件写入在事务提交后才发生——那么文件写入的失败意味着内存已经
  提交，无法回滚。这是需要解决的核心张力。

  观察 C：WAL 已经是某种形式的事务日志。 当前的 _wal_retry
  表实际上承担了"持久化意图日志"的角色。可以将这个机制升级为正式的事务日志。

  第六个问题：

  回滚是事务模型的核心能力。在我们的分层策略下，回滚在不同场景下有不同的难度：

  - 单文件 + 文件先写：文件写入失败 → 内存回滚无压力（什么都没丢）
  - 跨文件 + WAL 驱动：内存已提交 → 文件 A 写成功 → 文件 B
  写失败。此时内存无法回滚（已提交），只能靠 WAL 重试文件 B

  对于跨文件事务中"内存已提交但部分文件写失败"这个中间态，你倾向如何处理？

  A) WAL 最终一致 — 接受短暂的不一致窗口。内存中有新数据（用户可以看到），文件 B
  等 WAL 重试成功。查询走内存，用户无感知。重试全部失败 → 标记死信 →
  通知用户手动处理

  策略 B：预提交缓冲

  流程: 事务写入缓冲区 → 写文件A ✓ → 写文件B ✗ → 丢弃缓冲区 → 通知失败

  │ 致命缺陷   │ ❌ 文件 A 已写入，无法"撤回" — 丢弃缓冲区只能回滚内存，文件 │
  │            │  A 的修改已经落在磁盘上                                     │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 隔离       │ ⭐⭐⭐ 事务间完全隔离，未提交的修改不可见                   │

                                ↓
                           回滚内存事务
                                ↓
                           写补偿WAL: UNDO文件A
                                ↓

  ┌────────────┬─────────────────────────────────────────────────────────────┐
  │    维度    │                            评价                             │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 一致性模型 │ 最终一致，但保证回滚后的最终态是正确的（全部不变）          │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 回滚能力   │ ⭐⭐ 有 — 内存可回滚，文件 A 通过补偿 WAL 撤销              │

  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 补偿窗口   │ ⚠️ 文件 A 被改后到补偿 WAL 执行前，文件 A 处于"错误状态"    │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 崩溃场景   │ ⚠️ 如果在写补偿 WAL 之前崩溃 → WAL 中只有文件 A 的写入意图  │

  关键问题：补偿 WAL 本身也可能写入失败。而且补偿不是精确逆操作——策略 C
  背后其实是策略 A 的变种（依赖 WAL），只是多了一个"撤销意图"。

                        内存回滚      文件回滚       复杂度
  策略 A: WAL最终一致     ❌ 不能       ❌ 不能(只向前)   低

  场景: 单文件事务（90%+）
  策略: 文件先写 + 内存直接回滚

● 嵌套事务分析

  先看三个选项的语义

  场景: outer 中 inner 回滚，outer 继续

  // 期望: outer 不受影响
  outer: INSERT A → inner: INSERT B (失败回滚) → outer: INSERT C → commit
  结果: A ✅  C ✅  B ❌

  ---
  策略 A：合并到外层（Flat Nesting）

  实现: nestingLevel 计数器
    beginTx():  level++  (level=1 时才真正 BEGIN)
    commit():   level--  (level=0 时才真正 COMMIT)
    rollback(): 直接 ROLLBACK 整个外层(不管 level 多少)

  ┌──────────┬───────────────────────────────────────────────────────────────┐
  │   维度   │                             评价                              │
  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 实现     │ ⭐⭐⭐ 一个计数器即可                                         │
  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 函数组合 │ ⭐⭐⭐ 允许 — 防御性包装不伤大雅                              │
  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 部分回滚 │ ❌ 不支持 — inner rollback 炸掉整个 outer                     │
  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 惯用性   │ ⭐⭐⭐ PostgreSQL 就是这个行为（嵌套 BEGIN 被忽略，ROLLBACK   │
  │          │ 全回滚）                                                      │
  └──────────┴───────────────────────────────────────────────────────────────┘

  适用场景：函数 A 开事务调函数 B，B 内部防御性包装了事务。B 的 BEGIN/COMMIT
  是空操作，不破坏 A 的事务边界。

  陷阱：内部代码调了 rollback →
  外层所有工作白费。需要文档约定"不要在嵌套事务中调用 rollback"。

  ---
  策略 B：SAVEPOINT

  实现:
    beginTx():  当前是外层 → BEGIN; 当前已有事务 → SAVEPOINT sp_N
    commit():   当前是 savepoint → RELEASE sp_N; 当前是外层 → COMMIT
    rollback(): 当前是 savepoint → ROLLBACK TO sp_N; 当前是外层 → ROLLBACK

  ┌─────────────────┬───────────────────────────────────────────────────┐
  │      维度       │                       评价                        │
  ├─────────────────┼───────────────────────────────────────────────────┤
  │ 部分回滚        │ ⭐⭐⭐ 支持 — 内层失败不影响外层                  │
  ├─────────────────┼───────────────────────────────────────────────────┤
  │ SQLite 原生支持 │ ⭐⭐⭐ SAVEPOINT / RELEASE / ROLLBACK TO          │
  ├─────────────────┼───────────────────────────────────────────────────┤
  │ 实现            │ ⭐⭐ savepoint 管理栈                             │
  ├─────────────────┼───────────────────────────────────────────────────┤
  │ 复杂度          │ ⚠️ 栈深度、命名冲突、跨 savepoint 查询可见性      │
  ├─────────────────┼───────────────────────────────────────────────────┤
  │ 是否需要        │ ❓ MD-DB 真的有"内部操作失败但外层继续"的需求吗？ │
  └─────────────────┴───────────────────────────────────────────────────┘

  适用场景：批量导入数据时，每条记录一个
  savepoint，坏记录回滚+跳过，好记录继续。但 MD-DB
  的解析管道已经处理了这个——坏行在验证阶段就过滤了，不需要事务回滚。

  ---
  策略 C：禁止嵌套

  实现:
    beginTx(): 已有活跃事务 → throw Error("Transaction already active")

  ┌──────────┬───────────────────────────────────────────────────────────────┐
  │   维度   │                             评价                              │
  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 实现     │ ⭐⭐⭐ 一行代码                                               │
  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 函数组合 │ ❌ 差 — 被调函数无法防御性包装，需要额外传 inTransaction:     │

  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ 强制扁平 │ ⭐⭐⭐ 调用方必须把事务逻辑写平                               │
  └──────────┴───────────────────────────────────────────────────────────────┘

  问题：库函数不知道会被谁调用——视图层可能已经开了事务，库函数自己又想保证原子性

  综合对比

                  组合性   部分回滚   复杂度   惯用性
  策略 A: 合并     优       否        低      PostgreSQL 行为

  ---
  场景：转账操作

  // 一个库函数，作者防御性地加了事务包装
  function addExpense(accountId: string, amount: number) {
    engine.transaction(() => {           // 内部 BEGIN
      const acc = engine.select("accounts", { pk: accountId });
      const newBalance = acc.balance - amount;
      engine.update("accounts", { pk: accountId, balance: newBalance });
      engine.insert("transactions", { account: accountId, amount: -amount });
    });                                  // 内部 COMMIT
  }

  // 视图层调用：一次转两个账户
  engine.transaction(() => {             // 外层 BEGIN
    addExpense("alipay", 50);            // ← addExpense 内部又开了一个事务
    addExpense("wechat", 30);            // ← addExpense 内部又开了一个事务
  });                                    // 外层 COMMIT

  问题：addExpense 内部开了事务，外层也开了事务——这就是嵌套。怎么处理？

  ---
  策略 A：合并到外层（推荐）

  把嵌套调用当成一个大事务。

  时间线:
    engine.transaction(...)     → 真正执行 SQL: BEGIN
      addExpense("alipay",50)
        engine.transaction(...) → 计数+1，什么都不做（已在事务中）
          SELECT accounts       → 正常执行
          UPDATE accounts       → 正常执行
          INSERT transactions   → 正常执行
        transaction结束         → 计数-1，什么都不做（还没到最外层）
      addExpense("wechat",30)
        engine.transaction(...) → 计数+1，什么都不做
          SELECT accounts       → 正常执行
          UPDATE accounts       → 正常执行
          INSERT transactions   → 正常执行
        transaction结束         → 计数-1，什么都不做
    engine.transaction结束      → 真正执行 SQL: COMMIT

  一句话：无论嵌套多少层，最外层结束时才真正 COMMIT。内层的 BEGIN/COMMIT
  是空操作。

  rollback 的行为：任何人（内层或外层）调用 rollback → 全部回滚，不管嵌套几层。

  engine.transaction(() => {     // BEGIN
    addExpense("alipay", 50);    // 写入成功
    addExpense("wechat", 30);    // 内部调用 rollback → 全部回滚，alipay
  也白写了
  });                             // commit 不会执行

  ---
  策略 B：Savepoint

  每个内层事务创建一个"存档点"，回滚时只回到存档点。

  时间线:
    engine.transaction(...)     → SQL: BEGIN
      addExpense("alipay",50)
        engine.transaction(...) → SQL: SAVEPOINT sp_1
          UPDATE accounts       → alipay 余额扣了 50
          INSERT transactions   → 写入一条支出记录
        transaction失败(rollback)→ SQL: ROLLBACK TO sp_1  ← 只撤销 alipay 的修改
      addExpense("wechat",30)
        engine.transaction(...) → SQL: SAVEPOINT sp_2
          UPDATE accounts       → wechat 余额扣了 30
          INSERT transactions   → 写入一条支出记录
        transaction成功          → SQL: RELEASE sp_2      ← 保存 wechat 的修改
    engine.transaction结束      → SQL: COMMIT             ← wechat 生效, alipay
  没了

  结果：alipay 的 50 回滚了，wechat 的 30 成功了——部分回滚。

  ---
  策略 C：禁止嵌套

  直接抛错，不许嵌套。

  engine.transaction(() => {             // 外层事务已激活
    addExpense("alipay", 50);            // ← 内部调 transaction → 抛错!
    // Error: "Transaction already active"
  });

  addExpense 的作者必须这样写（不能自己包装事务）：

  function addExpense(accountId: string, amount: number) {
    // 不能再开事务！假设调用方已经开了
    const acc = engine.select("accounts", { pk: accountId });
    engine.update("accounts", { pk: accountId, balance: acc.balance - amount });
    engine.insert("transactions", { account: accountId, amount: -amount });
  }

  ---
  一句话总结

  ┌──────────────┬────────────┬─────────────┬───────────────┐
  │              │ 内层 BEGIN │ 内层 COMMIT │ 内层 ROLLBACK │
  ├──────────────┼────────────┼─────────────┼───────────────┤
  │ A: 合并      │   空操作   │   空操作    │   回滚全部    │
  ├──────────────┼────────────┼─────────────┼───────────────┤
  │ B: Savepoint │ 创建存档点 │ 释放存档点  │  只回滚内层   │
  ├──────────────┼────────────┼─────────────┼───────────────┤
  │ C: 禁止      │    抛错    │      —      │       —       │
  └──────────────┴────────────┴─────────────┴───────────────┘

  推荐的 A 方案
  就是：不管嵌套多少层，都当作一个大事务，最外层结束时统一提交。简单，够用。

● 推荐分析

  逐项判断

  CRUD 方法（A 的诉求）：必须的。 事务就是为了把多个 CRUD 打包。tx 上必须有
  insert、update、delete、select——否则事务对象没有任何操作能力。

  查询引擎（B 的诉求）：应该给。 典型场景：

  engine.transaction((tx) => {
    const balance = tx.query({
      table: "accounts",
      filter: { pk: accountId },
      aggregates: [{ fn: "sum", field: "amount" }]
    });
    if (balance < amount) {
      throw new Error("余额不足");  // 触发回滚
    }
    tx.insert("transactions", row);
  });

  事务中的查询应该和事务外的查询走同一套
  API，不需要区分。不给查询引擎意味着事务内只能用最原始的 select——功能残缺。

  tx.rollback()（C 的诉求）：不需要显式暴露。 回调中抛出异常 =
  回滚。这是最自然的语义：

  // 已有方式：抛异常 → 回滚
  engine.transaction((tx) => {
    tx.insert("a", row);
    if (someCondition) throw new Error("abort");  // 隐式回滚
  });

  // 不需要：
  engine.transaction((tx) => {
    tx.insert("a", row);
    if (someCondition) tx.rollback();  // 显式回滚——多余的API
  });

  throw 是 JavaScript 的标准流控机制，不需要发明新 API。而且 throw
  的优点是好调试——堆栈里能看到回滚原因。

  Savepoint（C 的诉求）：现在不需要。 我们已经确认了策略
  A（嵌套合并），不自动创建 savepoint。如果未来需要，加个 tx.savepoint()
  独立方法即可，不影响现在。

  ---
  结论：方案 B

  tx 上暴露的方法：
    CRUD:   tx.insert()  tx.update()  tx.delete()  tx.select()
    查询:   tx.query()   tx.queryRaw()
    回滚:   通过 throw 触发（不需要显式方法）

  interface Transaction {
    insert(table: string, record: Record): string;       // 返回 storage_pk
    update(table: string, pk: string, record: Partial<Record>): void;
    delete(table: string, pk: string): void;
    select(table: string, filter: SimpleFilter): Record[];
    query(q: Query): ResultSet;
    queryRaw(sql: string, params: any[]): ResultSet;
  }

  ---
  方案 A：操作累积（Buffer-First）

  tx.insert("t", row1)  → push { op: INSERT, table: "t", data: row1 }
  tx.update("t", pk, d) → push { op: UPDATE, table: "t", pk, data: d }
  tx.select("t", ...)   → 查 SQLite + 合并缓冲区（缓冲区的新数据覆盖 SQLite
  旧数据）
  commit()              → 验证冲突 → 生成文件 → vault.process()
                          ├─ 成功 → 重放缓冲到 SQLite → COMMIT
                          └─ 失败 → 丢弃缓冲 → 无事发生

  ┌──────────┬──────────────────────────────────────────────────────────────┐
  │   维度   │                             评价                             │
  ├──────────┼──────────────────────────────────────────────────────────────┤
  │ select   │ ⚠️ 需要合并逻辑 — 缓冲区中的 INSERT 还没进 SQLite，但 select │
  │ 的正确性 │  必须能看到。缓冲区 UPDATE 了某行 → 必须覆盖 SQLite          │

  │          │ 合并                                                         │

  │          │ ❌ 致命 — 表 A 的 INSERT 在缓冲区中 → 表 B 的 ref(A) 查询 A  │

  关键对比

  场景: tx.insert("a", row1); tx.select("a"); tx.insert("b", row2); commit

  方案 A:
    insert a → buffer: [{a, row1}]
    select a → SQLite(无row1) ∪ buffer → 手动合并返回 row1  ← 额外逻辑
    insert b → buffer: [{a, row1}, {b, row2}]
    commit   → 生成文件 → vault.process() → 重放到SQLite

  方案 B:
    insert a → SQLite INSERT(在事务中)  ← SQLite 原生
    select a → SELECT FROM a            ← 直接走 SQLite, 看到 row1
    insert b → SQLite INSERT(在事务中)
    commit   → 生成文件 → vault.process() → SQLite COMMIT

  1. 写一个"缓冲区合并层"夹在查询引擎和 SQLite 之间
  2. 或者事务内禁止使用 tx.query()，只允许简单的 tx.select()

  方案 B 中 tx.query() = engine.query()，完全复用，零额外代码。

  复杂度天平

                      方案 A              方案 B
  select 实现         复杂(合并层)         零(复用SQLite)
  query 复用          不能                 完全复用
  跨表 ref            不能(缓冲区隔离)      完美支持
  冲突检测实现        简单                 简单
  回滚实现            简单                 简单(SQLite ROLLBACK)
  文件写入            简单                 简单

  总代码量            ~200行 + 合并逻辑     ~50行(包装SQLite
  BEGIN/COMMIT/ROLLBACK)

  方案 A 唯一优势是"文件写入前不碰
  SQLite"，但这个优势在回滚成本上并没有体现——方案 B 的回滚同样是零成本的
  ROLLBACK。

  方案 B 就是 SQLite 的事务原生语义——我们只是在外层包了文件写入。它没有增加复杂

  engine.transaction((tx) => {

  选项 A:
    内存COMMIT → WAL[a]+WAL[b] 持久化 → [从此点之后, 任何崩溃都能恢复]
                  ↑
                 窗口 = 写WAL的时间(~1ms, 顺序写两个小文件)

  选项 B:
    内存COMMIT → [fileA ✓] → [fileB ✗] → WAL[b]
          ↑        ↑           ↑
          崩溃点1: 崩溃点2:     崩溃点3:
          全丢     永久分裂      WAL可恢复

  ┌─────────────────────┬─────────────────────────┬─────────────────────────┐
  │       崩溃点        │       选项 A 结果       │       选项 B 结果       │
  ├─────────────────────┼─────────────────────────┼─────────────────────────┤
  │ COMMIT 后立即崩溃   │ WAL 已有，重启重放 ✓    │ 内存丢，文件没变 →      │
  │                     │                         │ 数据丢失 ✗              │
  ├─────────────────────┼─────────────────────────┼─────────────────────────┤
  │                     │ WAL 已有                │ A 在文件上，B 丢失，WAL │
  │ fileA 写完后崩溃    │ A+B，重启重放（A        │  无记录 → 分裂 ✗        │
  │                     │ 幂等写入） ✓            │                         │
  ├─────────────────────┼─────────────────────────┼─────────────────────────┤
  │ fileB               │ WAL 已有，重启继续重试  │ WAL 已有，重启继续重试  │

  │ 写完了）            │                         │                         │
  └─────────────────────┴─────────────────────────┴─────────────────────────┘

  ---
  关键洞察

  选项 B 的 COMMIT 后崩溃是真正的数据丢失——事务已提交（调用方拿到结果），但数据
  永远丢了（内存无、文件无、WAL 无）。这在选项 A 中不存在——WAL 在 COMMIT
  之后、文件写入之前就持久化了。

  选项 B 的"正常路径零 WAL
  开销"是一个虚假优势——因为跨文件事务本身就是低频操作，为低频操作节省几毫秒的
  WAL 写入时间不值得，但为低频操作引入数据丢失风险不值得。

  ---
  建议：选项 A

  唯一原因：崩溃安全性。

  WAL 先写的本质是把"事务已提交"这个事实持久化到磁盘。在 Obsidian
  插件环境中，内存 SQLite 的 COMMIT 只存在于 WASM
  的堆内存中——进程退出就没了。如果没有 WAL
  把提交事实记录下来，崩溃就等于丢数据。

  选项 A 的流程图：

  engine.transaction((tx) => {       // 跨文件事务
    tx.insert("accounts", row1);     // → SQLite
    tx.insert("transactions", row2); // → SQLite
    return done;
  });

  //   1. SQLite COMMIT（内存提交）
  //   2. 生成 WAL 条目[fileA, fileB] → 持久化到 .obsidian/.../wal/

  // 如果步骤 2-4 之间崩溃:
  //   → 重启 → 读取 WAL 目录 → 重放所有条目 → 幂等写入文件

● WAL 条目格式分析

  先明确前提：跨文件事务中，SQLite 已提交，WAL
  的作用是记录"哪些文件还没同步"。数据本身已在 SQLite 中，WAL
  不需要存数据——只需要存"文件路径 → 表名"的映射。

  ---
  三种方案的 WAL 形态

  场景：跨文件事务修改了 accounts.md 和 transactions.md

  ---
  方案 A：保持行级 WAL

  沿用 Phase 10 的格式，每个操作独立一条：

  INSERT | accounts.md | storage_pk=acc:1:a3f7b2 |
  {"name":"支付宝","balance":950}
  UPDATE | transactions.md | storage_pk=tx:3:b1c4d5 | {"amount":-50}
  DELETE | transactions.md | storage_pk=tx:7:e2f8a3

  重放时逐条执行：INSERT → 末尾追加行，UPDATE → 按 storage_pk 定位行替换，DELETE
  → 按 storage_pk 删除行。

  ┌────────────────┬─────────────────────────────────────────────────────────┐
  │      维度      │                          评价                           │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ 重放正确性     │ ❌ 致命 — 重放时 storage_pk 中的 line_number            │
  │                │ 已失效。文件可能已被用户编辑，行 3 不再是原来的行 3     │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │                │ ❌ 困难 — INSERT                                        │
  │ 幂等性         │ 重复重放会在末尾追加重复行。需要查重逻辑（"这个         │
  │                │ storage_pk 已经存在了吗？"）                            │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ 事务边界       │ ❌ 丢失 — 3 条 WAL 之间没有关联。崩溃后重放到第 2       │
  │                │ 条时又崩溃 → 1 条执行了，2 条没执行 → 半事务状态        │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ 与文件先写冲突 │ ❌ 行级操作和                                           │
  │                │ vault.process()（整文件替换）是两种不同的写入模式       │
  └────────────────┴─────────────────────────────────────────────────────────┘

  核心问题：行级 WAL 是为"行级文件操作"设计的（逐行追加/替换/删除）。但跨文件事
  务中，我们更自然的做法是"从 SQLite
  重新生成整个文件"——不需要知道具体哪些行变了。

  ---
  方案 B：事务组 WAL

  引入 tx_id 分组：

  TX | tx_001 | accounts.md    | {"table":"accounts"}
  TX | tx_001 | transactions.md | {"table":"transactions"}

  重放时按 tx_id 分组：同一组的所有文件全部同步，才算该事务重放完成。

  ┌────────────┬─────────────────────────────────────────────────────────────┐
  │    维度    │                            评价                             │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 重放正确性 │ ⭐⭐⭐ 好 — 从 SQLite 重新生成文件，不依赖行号              │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 事务边界   │ ⭐⭐⭐ 清晰 — tx_id 明确标记了哪些文件属于同一事务          │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 幂等性     │ ⭐⭐⭐ 天然幂等 — 从 SQLite                                 │
  │            │ 生成文件内容，写入多少次结果都一样                          │
  ├────────────┼─────────────────────────────────────────────────────────────┤
  │ 部分恢复   │ ⚠️ 崩溃在写 WAL[a] 之后、WAL[b] 之前 → 只有 WAL[a] 被持久化 │
  │            │  → 重启只恢复 fileA → fileB 永久丢失                        │
  └────────────┴─────────────────────────────────────────────────────────────┘

  关键洞察：方案 B 的"写两个 WAL 条目"不是原子的。如果在写 WAL[a] 到磁盘后、写
  WAL[b] 前崩溃，fileB 的数据就丢了（SQLite 内存没了，WAL 中没有 fileB
  的记录）。

  方案 C：单条目事务 WAL

  整个事务一个 WAL 条目：

  {
    "tx_id": "tx_001",

      {"path": "finance/accounts.md", "table": "accounts"},

  ┌────────────┬────────────────────────────────────────────────────────────┐
  │    维度    │                            评价                            │
  ├────────────┼────────────────────────────────────────────────────────────┤
  │ 原子持久化 │ ⭐⭐⭐ 一个文件写入 =                                      │
  │            │ 原子。要么全部可恢复，要么全部丢失（不会半恢复）           │

  ├────────────┼────────────────────────────────────────────────────────────┤
  │ 幂等性     │ ⭐⭐⭐ 同 B                                                │

  │ WAL        │ ⭐⭐⭐ 极小 — 只有文件路径和表名，不含数据。100            │
  │ 条目大小   │ 个文件的条目也就几 KB                                      │
  ├────────────┼────────────────────────────────────────────────────────────┤
  │ 实现       │ ⭐⭐⭐ 最简单 — 不需要分组逻辑，不需要逐行重放             │
  └────────────┴────────────────────────────────────────────────────────────┘

  崩溃场景对比

  场景: 跨文件事务修改 fileA + fileB
        SQLite COMMIT 完成
        正在写 WAL...

  时间线:  [SQLite COMMIT] → [写WAL] → [写fileA] → [写fileB]

  方案 A:  写3条WAL(行级)
  方案 B:  写2条WAL(file级, 同tx_id)
  方案 C:  写1条WAL(事务级)

  ┌────────────┬────────────────────────┬──────────────┬───────────────────┐
  │   崩溃点   │         方案 A         │    方案 B    │      方案 C       │
  ├────────────┼────────────────────────┼──────────────┼───────────────────┤
  │ SQLite     │                        │              │                   │
  │ COMMIT     │ 全丢 ✗                 │ 全丢 ✗       │ 全丢 ✗            │
  │ 后、WAL    │                        │              │                   │
  │ 写入前     │                        │              │                   │
  ├────────────┼────────────────────────┼──────────────┼───────────────────┤
  │ 首条 WAL   │                        │ fileA        │ 文件损坏或不存在  │
  │ 写到一半   │ 部分恢复，行级不一致 ✗ │ 恢复，fileB  │ → 全丢(但一致) ✗  │
  │            │                        │ 丢失 ✗       │                   │
  ├────────────┼────────────────────────┼──────────────┼───────────────────┤
  │ 全部 WAL   │ 逐行重放(行号失效风险) │ 全部恢复 ✓   │ 全部恢复 ✓        │
  │ 写完       │  ✗                     │              │                   │
  └────────────┴────────────────────────┴──────────────┴───────────────────┘

  三种方案在"WAL 未写完就崩溃"时都有问题，但方案 C
  的窗口最小——只有一个文件写入操作。

  ---
  关键洞察

  方案 C 的 WAL 条目不含数据，只含元数据：

  数据在哪里？ → SQLite 中（已提交）
  WAL 存什么？ → 哪些文件需要从 SQLite 重新生成
  重放做什么？ → SELECT * FROM table → 生成文件内容 → vault.process()

  这就是"文件同步清单"模型——WAL 是待办列表，不是数据副本。

  ---
  建议：方案 C

  理由：

  1. 原子性：一个 JSON 文件写入 = 一个原子操作。不存在"半个事务"的可恢复状态
  2. 简洁：不需要 tx_id 分组逻辑，不需要行级重放逻辑
  3. 自然适配：重放 = 从 SQLite 重新生成文件，与正常写入路径完全一致
  4. WAL 大小不随事务规模增长：100 个操作的事务和 2 个操作的事务，WAL
