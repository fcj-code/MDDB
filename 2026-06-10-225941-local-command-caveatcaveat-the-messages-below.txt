> 分析建议                                                                      

  Thought for 21s (ctrl+o to expand)

● 让我分析三个选项的利弊。

  ---
  Option B（视图直连引擎）的问题

  每条读写路径都要将 QueryEngine 和 CRUD API 的调用缝合到视图组件中。每个视图类
  型（表、表单、后续的看板等）都会重复同样的管道：拼 Query → 调 query() → 解析
  ResultSet → 格式化 decimal/null/ref → 渲染；写入时又要自己调 engine →
  处理冲突/失败 → 刷新。重复 5 次就是 5
  套独立的读写管道，错误处理不统一，引擎接口变了要改 N 处。

  Option C（声明式数据绑定）的问题

  前期投资巨大。本质上是在代码 API 之上再建一层声明式框架 ——
  需要设计声明语法、解析器、action 路由、自动刷新机制。这是"当你有 50
  个视图且用户是终端用户"时才需要的抽象。我们现在的用户是开发者（写视图组件的我
  们自己），第一版就走这条路是过度设计。

  Option A（ViewModel）的理由

  视图层的一个核心事实是：读和写共享同一条数据路径。用户在表格中修改一个单元格：

  视图检测变更 → 乐观更新 UI → ViewModel.updateCell(pk, col, val)
    → 调引擎 update() → 处理冲突/失败 → 通知订阅者 → 其他视图感知变更

  ViewModel 是这个数据路径的"枢纽"。它不只是一个查询包装器 ——
  它是视图对引擎的唯一接触面。收益：

  - 每种视图类型只需写一次读写管道。TableViewModel 封装了表格特有的交互模式（排
  序、过滤、分页、单元格编辑、行删除）。FormViewModel
  封装了表单特有的交互（加载单记录、按字段保存、校验后提交）。后续加第 3、4
  种视图时，模式已经确立。
  - 格式化集中处理：decimal 显示、null 标记、ref 展开、默认排序 —— 在 ViewModel
  中处理一次，所有消费方受益。
  - 跨视图数据同步：两个视图看同一张表，一个改了数据，另一个通过 ViewModel
  的订阅自动感知。
  - 可测试性：ViewModel 可以 mock 掉引擎，独立测试其逻辑。

  建议：A

  这个是视图层的架构基础，一旦选定会影响后续所有视图的设计。你同意 A 吗？

● Option B（单一 ViewModel + 策略）否决

  所有视图类型塞进一个类，内部靠 if (viewType === 'table') ... else if (viewType
  === 'form') ... 分支。表格有分页，表单没有；表单有脏跟踪，表格没有。这些差异
  不是"换个策略对象"能优雅解决的，最终会变成一个上帝类。

  Option C（纯组合）的问题

  灵活但缺乏约束。每种新视图都可以自由选择组合哪些部件，两次实现可能完全不同。在
  "后期扩展更多视图时复用一套逻辑"的目标下，纯组合缺少强制性的公共层契约，容易写
  散。

  Option A（BaseViewModel 抽象基类）最匹配

  继承链恰恰是"复用一套逻辑"的实现手段：

  BaseViewModel
    每个视图必然需要的：                 每种视图自己定义的：
    ├ query(config)                    ├
  渲染适配数据（columns/rows、fields/values）
    ├ refresh()                        ├
  视图状态管理（sort/filter/pagination、dirty/validation）
    ├ subscribe(event, cb)             └
  视图特有操作（sort/filter/page、saveField/submit）
    ├ insert / update / delete
    ├ formatPipeline (decimal/null/ref)
    └ dispose()

  公共层定义所有视图一致的契约。加第三种视图（例如看板）时，只需实现它自己那部分
  。Base 层不会膨胀 —— 每种视图不同的东西不进 Base。

  一个关键约束：BaseViewModel
  只持有查询配置和事件总线，不知道"表格"或"表单"是什么。它不假设 UI
  形态。这保证后期加新视图类型时 Base 不用改。

  问题 3：BaseViewModel 的事件/订阅机制

  视图需要感知数据变化以更新 UI。场景包括：

  - 用户在表格视图修改单元格 → 同一页上的表单视图看到最新值
  - 外部文件变更（vault.on('modify')）→ 引擎重扫 → 视图刷新

  type ViewEvent =

    | { type: 'state-changed'; state: ViewState }    // 
  查询状态变更（sort/filter/page 变化）
    | { type: 'error'; errors: Error[] }             // 操作失败

  vm.on('error', (errors) => { ... })

  class BaseViewModel {
    onDataChange(cb: (changes: Change[]) => void): Disposable
    onStateChange(cb: (state: ViewState) => void): Disposable
    onInsert(cb: (record: Record) => void): Disposable
    onUpdate(cb: (pk: string, field: string, newValue: any) => void): Disposable
    onDelete(cb: (pk: string) => void): Disposable
    onError(cb: (errors: Error[]) => void): Disposable
  }

● 核心判断标准：跨视图同步场景

  一个表格视图编辑单元格后，旁边的表单视图需要感知。两种方案在这个场景下的表现：

  Option A（事件类型 + 负载内含语义）

  表格的 TableViewModel → 调基类 update() → 成功 →
    发 { type: 'data-changed', changes: [{ type: 'update', pk, field, newValue
  }] }
  表单的 FormViewModel 已订阅 'data-changed' → 收到通知 → 判断自己的 pk
  是不是被改动的 pk →
    是 → 刷新显示
    否 → 忽略

  一条订阅、一个处理函数、按需判断。

  Option B（细粒度回调）

  表单要关心 3 个回调：
    onInsert(cb) → 检查插入的记录是否自己正在编辑的
    onUpdate(cb) → 同上，更新显示
    onDelete(cb) → 同上，提示"记录已被删除"
  每个回调都要管理 Disposable。新增操作类型（如批量导入）→ 加新方法 →
  所有视图的订阅代码都要检查。

  表面更类型安全，实则把复杂性摊给了每个消费者。

  建议：A

  理由：

  - Obsidian 平台一致性：Obsidian 自己的 API 就是 vault.on('modify', cb)
  模式，保持一致减少学习成本
  - Change 类型用 discriminated union，TypeScript 的类型收窄仍然有效：

  type Change =
    | { type: 'insert'; record: Record }
    | { type: 'update'; pk: string; field: string; oldValue: any; newValue: any
  }
    | { type: 'delete'; pk: string }

  vm.on('data-changed', (changes) => {
    for (const c of changes) {
      switch (c.type) {
        case 'update': ... // TypeScript 知道 c.field, c.newValue
        case 'delete': ... // TypeScript 知道只有 c.pk
      }
    }
  })
  - API 表面小：on(type, cb) + off(type, cb) 两个方法，不随操作类型增长

  当前阶段没有终端用户写 Markdown 声明视图的需求 —— Phase 17

  的声明式查询入口（DQL，远期）是同一性质 —— 都该放在远期。

  Option A vs C 的关键区别

  实际上 C 的成本极低。如果 ViewModel 的配置对象本身就是一个纯数据的
  interface（可序列化），那 A 和 B 之间没有鸿沟：

  // 这就是一个纯数据对象 —— 既可以来自代码，也可以来自 YAML parse 结果
  interface TableConfig {
    table: string;
    columns?: string[];
    sort?: SortClause[];
    filter?: FilterGroup;      // 已经设计的结构，天然可序列化
    pageSize?: number;
  }

  // 编程式用法（当前）：直接构造配置对象
  const vm = new TableViewModel({ table: 'contacts', columns: [...], sort: [...]
  })

  // 声明式用法（远期）：解析 Markdown 代码块 → 同样的 JSON 配置
  // YAML 解析 → { table: 'contacts', columns: [...], sort: [...] } → new 
  TableViewModel(config)

  不需要"桥接层" —— 配置类型本身可序列化，序列化/反序列化是 JSON/YAML
  库的事，不是我们发明的。

  │                    │      └───────────────────┘
  │ ```mddb-form       │
  │ to transactions    │
  │ mode new           │
  │ ```                │
  └───────────────────┘

  用户在 home.md 的表单里录了一条「-56，餐饮」。

  ---
  策略 A：绑定代码块生命周期

  录入前：
    home.md表格的 vm → 查询引擎 → 得到50行数据
    home.md表单的 vm → 就绪，等待填写
    reports.md表格的 vm → 查询引擎 → 得到上个月的数据
    侧边栏的 vm → 查询引擎 → 得到本月合计

  用户切到 reports.md 看报表：
    home.md 两个代码块被卸载 → 两个 vm.dispose() → 释放在 home.md 的查询结果
    reports.md 代码块渲染 → 创建新 vm → 查询引擎 → 得到数据

  用户切回 home.md：
    reports.md 代码块卸载 → vm.dispose()
    home.md 两个代码块重新渲染 → 重新创建 vm → 重新查询引擎

  结果：每次切换笔记，ViewModel 都被销毁和重建。重新查询很快（数据在内存
  SQLite），但之前的 UI 状态（滚动位置、展开的列、选中行）全部丢失。

  ---
  策略 B：全局 ViewModel 池

  启动 Obsidian：
    → 后台扫描所有 md 文件 → 发现 transactions 表
    → 创建 transactions 的 BaseViewModel → 放入全局池 Map<table, ViewModel>
    → 这个 vm 一直在内存中，直到关闭 Obsidian

  home.md 渲染表格：
    → 从池中取出 transactions 的 vm
    → 但每个代码块有不同的 show/sort/where → 用谁的状态？

  这里出了矛盾：池中只有一个 transactions 的
  ViewModel，但三个地方用不同的过滤、列、排序。A 说"显示 50 条最近记录"，B
  说"只看 6 月"，C 说"只要合计数字"。全局池的单个 ViewModel
  无法同时满足三个不同查询。

  要么让池存多份（每个代码块一个），但这和策略 A
  没区别了。要么强制全局共享同一个 ViewModel
  状态，但三个视图看到的数据就会互相干扰。

  ---
  策略 C：数据层全局池化 + 视图层随代码块

  启动 Obsidian：
    → DataLayer 池化（内存缓存，全局一份）：
        · 查询结果缓存：transactions 的全量 SQLite 数据已经可用
        · 事件总线：任何位置改 transactions → 广播 data-changed
        · 格式化器：deciaml/null/ref 处理
    → UI 层随代码块（每个块独立）：
        · 自己的 TableConfig（show/sort/where 各不相同）

  home.md 表格 vm：
    UI 层：config = { limit: 50, sort: 日期 desc }  ← 自己的
    数据层：从全局事件总线订阅 data-changed           ← 共享的

  home.md 表单 vm：
    UI 层：config = { mode: 'new', fields: [...] }   ← 自己的
    数据层：同一个事件总线                              ← 共享的

  用户录入一笔新记录：
    表单 vm → engine.insert() → 数据层广播 "insert 了一笔"
    → 同一页的表格 vm 收到 → refresh() → 表格里立刻出现新行
    → reports.md 还没打开（已卸载），但下次打开时会从数据层拿到最新全量

  用户切到 reports.md：
    home.md 两个 vm 卸载 → UI 层销毁（排序状态、列选择、滚动位置消失）
    数据层还在 → 事件总线还在
    reports.md 表格 vm 创建 → 从数据层拿数据 → 立刻渲染（没延迟）
