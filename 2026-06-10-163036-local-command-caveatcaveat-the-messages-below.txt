  ---
  问题 5：查询执行模型——同步还是异步？

  数据在内存 SQLite（sql.js/WASM），查询不涉及 IO。但有两个变量：

  ┌──────────────────────────┬──────────────────────────────────┐
  │         考虑因素         │             实际情况             │
  ├──────────────────────────┼──────────────────────────────────┤
  │ 数据都在内存             │ 无磁盘 IO，同步即可              │
  ├──────────────────────────┼──────────────────────────────────┤

  选项: A: 同步
  做法: query(q): ResultSet，直接返回

● 第一节：查询对象 (Query) 核心结构

  interface Query {
    table: string;                    // 表名，必填
    select?: SelectClause;            // 选择哪些列，默认全部
    filter?: FilterGroup;             // WHERE 条件，默认无
    sort?: SortClause[];              // ORDER BY，默认按 @sort 或写入时序
    limit?: number;                   // 返回行数上限
    offset?: number;                  // 偏移量
    groupBy?: string[];               // GROUP BY 列名列表
    having?: FilterGroup;             // HAVING 条件（配合 groupBy）
    aggregates?: AggregateClause[];   // 聚合函数
    followRefs?: FollowRefClause[];   // ref 字段跟随展开
    raw?: string;                     // SQL escape hatch（与上述结构互斥）
  }

  几个关键点：

  - raw 存在时忽略所有结构化字段，直接透传 SQL 给 sql.js
  - filter 用嵌套 FilterGroup 表达 AND/OR 树，这是整个结构最复杂的部分
  - followRefs 是场景 4 的核心——查询 transactions 时自动把 账户 字段对应的
  accounts 表记录展开

  先确认框架对吗？然后我展开最核心的三个子结构：FilterGroup（过滤）、AggregateCl
  ause（聚合）、FollowRefClause（关联展开）。

  第二节：FilterGroup（过滤条件树）

  // 单个条件
  interface FilterCondition {
    field: string;                              // 列名
    op: '=' | '!=' | '>' | '<' | '>=' | '<='   // 比较
      | 'in' | 'not_in'                          // 集合
      | 'like' | 'not_like'                      // 模糊匹配
      | 'is_null' | 'is_not_null';               // 空值判断
    value?: any | any[];                         // is_null / is_not_null 不需要
  value
  }

  // 条件组——递归结构，表达 AND/OR 嵌套
  interface FilterGroup {
    logic: 'and' | 'or';               // 组内条件的逻辑关系
    conditions: (FilterCondition | FilterGroup)[];  // 可嵌套
  }

  示例——简单过滤：

  // WHERE 分类 = '餐饮' AND 金额 < -50
  {
    logic: 'and',
    conditions: [
      { field: '分类', op: '=', value: '餐饮' },
      { field: '金额', op: '<', value: -5000 }  // decimal 存整数
    ]
  }

  示例——嵌套：

  // WHERE (分类 = '餐饮' OR 分类 = '交通') AND 金额 < -50
  {
    logic: 'and',
    conditions: [
      {
        logic: 'or',
        conditions: [
          { field: '分类', op: '=', value: '餐饮' },
          { field: '分类', op: '=', value: '交通' }
        ]
      },
      { field: '金额', op: '<', value: -5000 }
    ]
  }

● 第三节：聚合

  interface AggregateClause {
    field: string;                    // 聚合的列名，'*' 表示 COUNT(*)
    fn: 'count' | 'sum' | 'avg' | 'min' | 'max';
    alias?: string;                   // 结果列名，默认自动生成如 'sum_amount'
  }

  示例——按分类统计支出：

  // SELECT 分类, SUM(金额) as total, COUNT(*) as cnt
  // FROM transactions
  // WHERE 金额 < 0
  // GROUP BY 分类
  // HAVING total < -10000

  const q: Query = {
    table: 'transactions',
    filter: { logic: 'and', conditions: [
      { field: '金额', op: '<', value: 0 }
    ]},
    groupBy: ['分类'],
    aggregates: [
      { field: '金额', fn: 'sum', alias: 'total' },
      { field: '*', fn: 'count', alias: 'cnt' }
    ],
    having: {
      logic: 'and',
      conditions: [
        { field: 'total', op: '<', value: -10000 }  // 引用 alias
      ]
    }
  };

● 第四节：Ref 跟随展开

  这是 MD-DB 查询中最有特色的部分。回顾场景：transactions 表有 账户 字段类型为
  ref(accounts)，存储的是 accounts 表的 logical_pk
  值。查询时想看到账户名而不仅是 PK 值。

  interface FollowRefClause {
    field: string;              // 当前表中的 ref 字段名，如 '账户'
    include?: string[];         // 目标表的列名，默认全部，如 ['账户名', '类型']
    prefix?: string;            // 结果列前缀，默认目标表名，如 'accounts'
  }

  示例：

  // 查询交易记录，展开关联账户信息
  const q: Query = {
    table: 'transactions',
    filter: { logic: 'and', conditions: [
      { field: '日期', op: '>=', value: '2024-06-01' }
    ]},
    sort: [{ field: '日期', dir: 'asc' }],
    followRefs: [
      { field: '账户', include: ['账户名', '类型'], prefix: 'acc' }
    ],
    limit: 20
  };

  引擎内部执行（2 步）：

  Step 1: SELECT * FROM transactions WHERE 日期 >= '2024-06-01' LIMIT 20
          → 得到结果集，其中 acc 列值为 logical_pk

  Step 2: 收集所有 ref 值 → SELECT storage_pk, logical_pk, 账户名, 类型
          FROM accounts WHERE logical_pk IN (vk1, vk2, ...)
          → 按 prefix 展开: results[i].acc_账户名, results[i].acc_类型

  ResultSet 中的体现：

  // 返回的 columns 增加展开列
  columns: [
    { name: '日期', type: 'date' },
    { name: '金额', type: 'decimal' },
    { name: '账户', type: 'ref' },          // 原始 ref 值保留
    { name: 'acc_账户名', type: 'string' },   // 展开列
    { name: 'acc_类型', type: 'string' },     // 展开列
    ...
  ]

  关键决定：
  - 原始 ref 值保留，展开列追加——不会覆写
  - 采用两步查询而非 SQL JOIN：数据可能跨文件，绑定表解析更可靠
  - 引用断裂：目标不存在 → 展开列值为 NULL，不报错

  这个可以吗？
